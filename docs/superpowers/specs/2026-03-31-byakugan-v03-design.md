# Byakugan v0.3 — Design Spec

**Date:** 2026-03-31
**Status:** Approved
**Scope:** Full orchestration layer — superpowers integration, unified DB, GitFlow enforcement, unified hook engine

---

## 1. Goal

Byakugan v0.3 becomes the project-level orchestration layer that wraps and configures superpowers, adds what superpowers lacks (persistent project memory, session continuity, stack intelligence, GitFlow enforcement, model selection), and maintains all persistent state in a single local database that is never pushed to remote.

Superpowers handles workflow discipline. Byakugan handles everything else.

---

## 2. Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│                    GLOBAL (~/.claude/)                       │
│  settings.json → superpowers SessionStart hook              │
│  plugins/ → superpowers skills (untouched by Byakugan)      │
└─────────────────────────────────────────────────────────────┘
                            ↕ Claude Code merges
┌─────────────────────────────────────────────────────────────┐
│                 PROJECT (.claude/ + .byakugan/)              │
│  settings.local.json → Byakugan hooks (3 hooks)             │
│  CLAUDE.md → superpowers integration + GitFlow + models     │
│  .byakugan/byakugan.toml → project config                   │
│  .byakugan/byakugan.db → unified SQLite DB                  │
│  .byakugan/skills/ → GitFlow + model-selection skills       │
└─────────────────────────────────────────────────────────────┘
```

Byakugan never modifies `~/.claude/settings.json`, never touches the superpowers plugin directory, and never interferes with superpowers' own hook or skill execution. The two systems coexist additively at different configuration scopes.

## 3. Superpowers Integration

### 3.1 Detection & Installation

`byakugan init` runs this flow before any other setup step:

1. Check `~/.claude/plugins/cache/claude-plugins-official/superpowers/` for an installed version
2. Found → read version, verify `SessionStart` hook present in `~/.claude/settings.json`, proceed
3. Not found → run `claude plugin install superpowers`, verify success, warn if `claude` CLI unavailable

Byakugan records whether it installed superpowers in `byakugan.toml` (`superpowers_installed_by_byakugan = true`). On `byakugan deinit`, if this flag is set, user is offered the option to uninstall superpowers.

### 3.2 Byakugan-Owned Skills

Two skill files written to `.byakugan/skills/` during `init`:

**`gitflow-workflow.md`** — see Section 5
**`model-selection.md`** — see Section 6

These are **not** invoked via the `Skill` tool (which only loads from globally installed plugins). Instead, their full content is embedded directly as sections in CLAUDE.md — they are rule sets and reference tables, not interactive workflow skills. The `.byakugan/skills/` files are the canonical source; `byakugan update` re-embeds them into CLAUDE.md when they change.

### 3.3 CLAUDE.md Structure

Generated by Byakugan. Never generated by superpowers. Structure:

```markdown
<!-- byakugan:managed -->

## Workflow
[superpowers skill chain: using-superpowers → brainstorming → writing-plans →
 subagent-driven-development / executing-plans → finishing-a-development-branch]
[model selection rules summary + reference to model-selection skill]

## GitFlow
[gitflow skill reference]
[branch rules summary]
[hard rules: no AI mentions in git artifacts]

## Project Memory
[how to use byakugan remember, byakugan handoff]
[memory types + importance rules]
[never store ephemeral info — use handoff for session state]

## Active Guidelines
[table of .byakugan/ template files with paths and purposes]

## Project Context
[detected stack: languages, frameworks, tools]

## Privacy — Hard Rules
[never commit .byakugan/, CLAUDE.md, .claude/settings.local.json,
 .claude/remember/, .remember/, docs/superpowers/, byakugan.toml, *.db]
[GitFlow skill verifies staged files before every commit]
```

## 4. Unified Database (`byakugan.db`)

Replaces `memory.db`. Single SQLite file at `.byakugan/byakugan.db`.

### 4.1 Schema

```sql
-- Project knowledge (evolved from existing memory.db)
CREATE TABLE memories (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at       TEXT    NOT NULL,
    type             TEXT    NOT NULL,  -- correction|decision|preference|pattern|note
    content          TEXT    NOT NULL,
    context          TEXT,              -- JSON: {language, file, ...}
    tags             TEXT,              -- JSON array
    importance       INTEGER NOT NULL DEFAULT 3,  -- 1-5
    last_surfaced_at TEXT,
    surface_count    INTEGER NOT NULL DEFAULT 0
);

-- Raw tool-use events (current session buffer)
CREATE TABLE session_events (
    id               INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id       TEXT    NOT NULL,
    captured_at      TEXT    NOT NULL,
    tool_name        TEXT    NOT NULL,
    file_path        TEXT,
    exchange_summary TEXT    NOT NULL,
    tokens_estimate  INTEGER NOT NULL DEFAULT 0
);

-- Haiku-compressed history (hourly → daily → weekly)
CREATE TABLE session_summaries (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id          TEXT    NOT NULL,
    period              TEXT    NOT NULL,  -- hourly|daily|weekly
    created_at          TEXT    NOT NULL,
    content             TEXT    NOT NULL,
    source_event_count  INTEGER NOT NULL DEFAULT 0
);

-- Manual handoff notes (one active at a time)
CREATE TABLE session_handoffs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at  TEXT    NOT NULL,
    content     TEXT    NOT NULL,
    branch      TEXT,
    active      INTEGER NOT NULL DEFAULT 1  -- boolean
);
```

### 4.2 Compression Pipeline

```
session_events (raw)
  → [threshold: 50+ events OR 1hr elapsed]
  → Haiku compress (background subprocess)
  → session_summaries (period=hourly)
  → session_events pruned

session_summaries (hourly, end of day)
  → Haiku compress
  → session_summaries (period=daily)
  → hourly summaries pruned

session_summaries (daily, end of week)
  → Haiku compress
  → session_summaries (period=weekly)
  → daily summaries kept for 7 days then pruned
```

Compression is always async (background subprocess). Never blocks a hook response.

### 4.3 Migration

`byakugan init` on a project with existing `memory.db`:
1. Creates `byakugan.db` with full schema
2. Migrates all rows from `memories` table (schema-compatible, direct copy)
3. Renames `memory.db` → `memory.db.bak` (kept for 30 days, then `byakugan doctor` flags it)
4. Updates `byakugan.toml` to reference new DB path

## 5. GitFlow Skill (`gitflow-workflow.md`)

### 5.1 Branch Model

```
main       — production only, protected, never commit directly
develop    — integration branch, never commit directly
feature/… → PR to develop
release/x.x.x → PR to main + tag + back-merge to develop
hotfix/…   → PR to main + tag + back-merge to develop
```

### 5.2 Developer Discipline Rules

Claude Code behaves as a disciplined developer at all times:

**Before any file edit:**
- Verify current branch (`git branch --show-current`)
- If on `main` or `develop`: stop, create or switch to correct feature/hotfix branch first
- Verify branch is up to date with develop (`git fetch && git status`)
- If behind remote: pull + rebase before starting work

**Commits:**
- One atomic commit per logical unit of work — never batch unrelated changes
- Conventional Commits format: `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:`, `perf:`
- Commit message body explains *why*, not *what*
- No AI mentions anywhere: no `Co-Authored-By: Claude`, no `🤖 Generated with`, no "AI-assisted" in any git artifact (commits, PR titles, PR bodies, comments, tags, branch names)
- Never skip hooks (`--no-verify` forbidden unless user explicitly requests)
- Never amend published commits

**Pull Requests:**
- PR title: short, imperative, human-written
- PR body: scope, motivation, test plan — written as a developer would write it
- No AI footers, no AI signatures
- Target: `develop` (never `main` directly, except hotfix/release)

**Working tree discipline:**
- Stash before branch switches if working tree is dirty
- Clean working tree before PR
- Squash fixup commits before PR if history is noisy (ask user first)

**When unsure, ask:**
- "Should this be a feature branch or a hotfix?"
- "Should I squash these commits before the PR?"
- "This touches both auth and billing — one branch or two?"
- "The branch is 3 commits behind develop — should I rebase first?"

**Hard blocks (never proceed without resolution):**
- Staged files include `.byakugan/`, `CLAUDE.md`, `settings.local.json`, `*.db`, `docs/superpowers/` → unstage, warn, explain
- Committing to `main` or `develop` directly → abort
- Force-pushing → abort unless user explicitly confirms and explains why

## 6. Model Selection Skill (`model-selection.md`)

### 6.1 Rules

| Task | Model |
|------|-------|
| Brainstorming, design, architecture decisions | Opus 4.6 |
| Writing implementation plans | Opus 4.6 |
| Code review dispatch (spec + quality reviewers) | Opus 4.6 |
| Systematic debugging phases 1–2 (root cause, pattern analysis) | Opus 4.6 |
| Implementation, TDD cycles, refactoring | Sonnet 4.6 |
| Verification, unit tests, linting fixes | Sonnet 4.6 |
| Systematic debugging phases 3–4 (hypothesis + fix) | Sonnet 4.6 |
| DB compression (internal only) | Haiku 4.5 |

### 6.2 Enforcement

**Subagent dispatch:** Skills that dispatch subagents specify model explicitly using the `model` parameter. Brainstorming/planning → Opus. Implementation/verification → Sonnet.

**Interactive sessions:** When entering a brainstorming or planning phase, the pre-tool hook emits a reminder: `[byakugan] planning phase detected — consider switching to Opus 4.6 for this step`. When entering implementation, reminder to switch to Sonnet if on Opus.

## 7. Unified Hook Engine

### 7.1 Entry Point

Single Python entry point replacing `hook_runner.py`:

```
byakugan hook --type session-start   ← SessionStart
byakugan hook --type pre-tool        ← PreToolUse
byakugan hook --type post-tool       ← PostToolUse
```

### 7.2 session-start Handler

Output budget: ≤ 800 tokens. Priority order if over budget:

1. Active handoff note (≤ 150 tokens) — always included
2. Today's hourly summaries (≤ 300 tokens)
3. Top 5 high-importance memories, importance 4–5 (≤ 200 tokens)
4. Recent weekly digest (≤ 150 tokens) — dropped first if over budget

If compression is overdue (events pending, last compression >1hr ago): spawns background compression subprocess before emitting context.

### 7.3 pre-tool Handler

Same routing logic as current `hook_runner.py`, plus:
- Emits current branch name in header
- Warns if on protected branch (`main`, `develop`)
- Emits model-selection reminder if tool pattern matches planning/brainstorming context
- Budget: ≤ 300 tokens

### 7.4 post-tool Handler

Runs async (non-blocking):
1. Write raw event to `session_events`
2. Check compression threshold (50+ uncompressed events OR 1hr since last run)
3. If threshold met: spawn background subprocess for Haiku compression
4. Return immediately

### 7.5 settings.local.json

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "startup|clear|compact",
      "hooks": [{"type": "command", "command": "byakugan hook --type session-start"}]
    }],
    "PreToolUse": [{
      "matcher": "Edit|Write|MultiEdit|Bash",
      "hooks": [{"type": "command", "command": "byakugan hook --type pre-tool"}]
    }],
    "PostToolUse": [{
      "matcher": "Edit|Write|MultiEdit|Bash",
      "hooks": [{"type": "command", "command": "byakugan hook --type post-tool", "async": true}]
    }]
  }
}
```

## 8. New & Changed CLI Commands

### Changed

**`byakugan init`**
- Step 1: detect/install superpowers (new)
- Step 2: migrate `memory.db` → `byakugan.db` if found (new)
- Step 3: stack detection + template selection (unchanged)
- Step 4: write `.byakugan/skills/gitflow-workflow.md` + `model-selection.md` (new)
- Step 5: generate new CLAUDE.md format (updated)
- Step 6: write all three hooks to `settings.local.json` (updated from one hook to three)
- Step 7: update `.gitignore` comprehensively (updated)

**`byakugan deinit`**
- Removes all three hooks (updated)
- Offers to uninstall superpowers if Byakugan installed it (new)

**`byakugan doctor`**
- Adds: superpowers installation check
- Adds: all three hooks present check
- Adds: DB integrity check (schema, no corruption)
- Adds: compression backlog check (warns if >200 uncompressed events)
- Adds: `memory.db.bak` cleanup flag (after 30 days)

**`byakugan memories prune`**
- Also prunes `session_events` already captured in summaries

### New

**`byakugan handoff "<note>"`**
- Deactivates previous handoff, writes new record to `session_handoffs`
- Surfaced first in next session-start output
- Equivalent of `/remember` skill from the remember plugin

**`byakugan session`** subcommand group:
- `save` — manually trigger Haiku compression now (useful before `/clear`)
- `list` — show session summaries (period, date, preview)
- `show <id>` — full summary content

### Unchanged

`add`, `remove`, `update`, `sync`, `status`, `list`, `remember`, `memories list/search/edit/forget`

## 9. Privacy & `.gitignore`

### 9.1 Entries Managed by Byakugan

```gitignore
# Byakugan — never push
.byakugan/
CLAUDE.md

# Claude Code local config — never push
.claude/settings.local.json
.claude/remember/

# Session memory — never push
.remember/

# Superpowers / AI tooling — never push
docs/superpowers/
```

### 9.2 Enforcement

- `byakugan init`: adds missing entries (idempotent, never overwrites)
- `byakugan doctor`: verifies all entries present, auto-repairs if missing
- GitFlow skill: checks staged files before every commit, aborts if any privacy-protected path is staged, unstages automatically, explains why

## 10. Token Budget Summary

| Hook | Budget | Notes |
|------|--------|-------|
| session-start | ≤ 800 tokens | Drops weekly digest if over budget |
| pre-tool | ≤ 300 tokens | Same as current, well-tuned |
| post-tool | 0 tokens | Async, never adds to context |
| Haiku compression | ~$0.001–0.005/day | Background only |

Skill loading remains lazy — superpowers and Byakugan skills only load when invoked by name via the `Skill` tool.

## 12. Out of Scope

- Web UI or dashboard
- Team-shared memory (all memory is per-developer, per-project)
- Remote sync or cloud backup of DB
- Modification of superpowers skills or global Claude Code config
- Any telemetry or usage tracking
